Final project

 

Design, Model and Build a Database for the Capstone Project.

My goals for the project were to build a Capstone Project Database that would enable the development phase to begin. I also intended to learn how to use Rational Rose Data Modeler.

The Project refers to an application that will run various functions for a manufacturing or distribution business. I chose such a large scope because the important issues for me are to work the technologies, not the application per se.

The MSIE program for me is about learning the skills and gaining some experience working with the latest technologies, in particular J2EE and Web Services. I need to understand what works, what doesn't work, what works well and what works poorly. I need to learn how to design solutions based on these technologies. I hope to create a set of best practices that will be the foundations of my work for many years into the future.

The database model I have created consists of some 46 tables, which is certain to increase as development gets underway. While the traditional method of designing by drawing on very large pieces of paper were useful, it was also found to be wanting. Frequent changes to the data objects and their relationships caused far too much rework. The thinking process was very much impeded by the time-consuming process of redrawing what I had just envisioned. Even when drawn, the relationships of the objects were not clear.

I created objects with a SQL script using vi. This works well as it is possible to create objects very quickly. The scripts can be executed very quickly, errors determined and fixes made. However, the script in no way suggests anything about the objects, and says not a thing about their relationships. While this limitation may be acceptable for a relatively small number of objects, I found, with this project, these problems to be extremely problematic.

I tried to create data objects in Rational Rose Data Modeler. These objects are created graphically which is modern, but actually quiet slow. I could draw the data object relationships graphically, but it was not entirely obvious as to the object attributes being referenced.

I found the best practice was to create the objects in a SQL script, run the scripts to ensure that Oracle at least creates the objects without errors, and then to reverse engineer the script into Rational Rose Data Modeler. I then created Class Diagrams, dragging tables onto the diagram so the resulting diagram became of immense value in understanding the data. I made changes to the model, and then made changes to the scripts, running around the loop until the result was satisfactory. Although I could forward engineer the SQL script from the Rational Rose Data Modeler, I chose not to as the resulting SQL script was very poor in terms of readability.

Used in this manner, the tool is extremely powerful, easy to use and highly suggestive as to errors in the model.

Care must be taken with the choice of data types. I found that Oracle's facility to create your own data types is simply not supported in Rational Rose. Also, although Rational Rose allows you to select an Oracle 9x database, it actually is not supported (ref: rational Rose Technical Support). Oracle 8x is supported, but I found it was best to avoid any non-generic data structures.

The database model I have created will doubtless change but it is a good start.

Other

I did not create rows because I felt it not worth the effort at this point. Once development gets underway, the requirements will be much clearer, and then I will create some data.

I did not create any views because it is not obvious which views I will need.

I did not create delete triggers as the data model is not settled enough for it to be worth the effort.

Could create a Unix script to read a SQL script forward engineered from Rational Rose and write a much more readable SQL script.

Rational Rose uses comments from a reverse engineered script and applies them to the data objects. I need to learn how to make much more use of this feature.
